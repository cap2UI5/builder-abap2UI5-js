# AGENTS.md — builder-abap2UI5-js

Guidance for AI agents and contributors. Single source of truth for this
repository's structure and rules. Read before making any change.

## The cap2UI5 ecosystem (where this repo fits)

cap2UI5 brings the [abap2UI5](https://github.com/abap2UI5/abap2UI5) concept
to CAP/Node.js. Since the monorepo split (2026-07) it is organized as six
repositories under the `cap2UI5` org:

| Repo | Role | Generated? |
|---|---|---|
| **builder-abap2UI5-js** (this repo) | framework build: ABAP→JS transpiler, sync pipelines, generates the core package `abap2UI5` into `core/`; hosts the platform adapters | `core/` and `run/` are generated |
| [builder-cap2UI5](https://github.com/cap2UI5/builder-cap2UI5) | CAP app build: assembles `src/` + the published core, publishes 1:1 into the app repo | `run/input/core` is a mirror |
| [cap2UI5](https://github.com/cap2UI5/cap2UI5) | the finished, deployable CAP app | fully generated (except `.github/`) |
| [builder-cap2UI5-web](https://github.com/cap2UI5/builder-cap2UI5-web) | browser build tooling (bundles the app for GitHub Pages) | tooling only |
| [web-cap2UI5-build](https://github.com/cap2UI5/web-cap2UI5-build) | the built static site, served via GitHub Pages | fully generated |
| [docs](https://github.com/cap2UI5/docs) | VitePress documentation site | hand-written |

Data flow: `abap2UI5/abap2UI5` + `abap2UI5/samples` (upstream ABAP) →
**this repo** (mirror → transpile → assemble → publish `core/`) →
builder-cap2UI5 (`mirror_core` → assemble app, vendor core → publish) →
cap2UI5 app repo → builder-cap2UI5-web (mirror → bundle) →
web-cap2UI5-build → GitHub Pages.

Trigger chain (each hop = a bot commit of `UPSTREAM_HEAD` or a publish via
SSH deploy key; workflows skip gracefully when the secret is unset):
`trigger_cap` here → builder-cap2UI5 (`ACTION_KEY_CAP`) →
`update_cap` there publishes to cap2UI5 (`ACTION_KEY_APP`) →
`trigger_web` in cap2UI5 → builder-cap2UI5-web (`ACTION_KEY_WEB`) →
`build web` there pushes to web-cap2UI5-build (`BUILT_DEPLOY_KEY`).

## Layout

| Path | What it is | Hand-edit? |
|---|---|---|
| `src/` | hand-written SOURCE of the core package: `package.json` (npm name `abap2UI5`, the `exports` map), engine + ports, hand-ported classes, `README.md` (published as the core's README) | **yes — this is where framework changes go** |
| `core/` | the published package `abap2UI5` (src + generated overlays), committed | **never** — regenerated by `npm run build_core` |
| `run/input/{abap2UI5,app,samples}` | 1:1 upstream mirrors (committed, with `UPSTREAM_COMMIT`) | never — rewritten by the mirror steps |
| `run/output/{abap2UI5,samples,app}` | transpiled/prepared trees (committed) | never |
| `run/output/core` | assembled package staging (gitignored) | never |
| `run/output/tests` | transpiled upstream unit tests (gitignored scratch) | never |
| `adapters/{cap,node,express,web}` | platform adapters consuming `core/` via `file:../../core` | yes |
| `scripts/` | the build scripts (see below) | yes |
| `test/` | jest suite + known-failures ratchets | yes |

## Build & test

```bash
npm install                      # dev deps: @abaplint/core, jest
(cd adapters/cap && npm install) # backs the assemble load-gate (@sap/cds)
npm run build_core               # assemble_core + publish_core → core/
npm test                         # 17 suites / ~220 tests
```

Pipeline steps (all runnable standalone, `MIRROR_SOURCE=/path` uses a local
checkout instead of cloning):
`mirror_abap2ui5|mirror_app|mirror_samples` → `transpile_abap2ui5|transpile_samples|prepare_app`
→ `assemble_core` → `publish_core`.

Assemble policies (scripts/assemble-core.js): `src/` wins on every conflict;
backend overlay is add-only + **load-gated** (each fill-in must `require()`
— third-party deps resolve via `adapters/cap/node_modules` through
NODE_PATH); samples are flattened into `core/srv/app/samples/`; the webapp
replaces `core/app/z2ui5/webapp` 1:1. Non-parsing transpiles are skipped and
reported.

Test ratchets: `test/upstream-units.known-failures.json` and
`test/apps-smoke.known-failures.json` — a new failure is a regression (red),
a fixed entry must be delisted (red). `node scripts/ratchet-update.js`
reconciles both baselines (delists fixed entries, fails on regressions;
`--check` for a dry run) — the `build_core` pipeline runs it and commits the
shrunken baselines, so only regressions can turn the nightlies red.
`scripts/oracle-classify.js` classifies the baseline against the upstream
`node/` runtime (official @abaplint transpiler + open-abap): entries green
there are provable-in-JS; the remaining ones here need ABAP type/reference
semantics our idiomatic port deliberately avoids (scalar/dref identity,
RTTI-typed conversions, sync-over-async) — reasons live in the baseline.

## Workflows (.github/workflows)

- `update_samples` (03:00) / `update_backend` (04:00) / `update_frontend`
  (05:00): nightly `mirror → transpile/prepare → build_core` pipelines;
  `build_core` runs jest and only commits `core/` on green.
- `trigger_update`: all three back-to-back (manual).
- `test`: jest on PRs / pushes to main.
- `trigger_cap`: manual; writes the sha of the last commit touching `core/`
  into builder-cap2UI5:`UPSTREAM_HEAD` (deploy key `ACTION_KEY_CAP`) to kick
  the downstream CAP build.

## Rules

- Never hand-edit `core/` or anything under `run/` — edit `src/` (or the
  transpiler in `scripts/abap2js.js`) and rebuild. `scripts/abap2js.js` is a
  thin re-export of the transpiler modules under `scripts/abap2js/` (lex,
  naming, model, expr, stmt, method, interface, index, cli).
- `core/README.md` comes from `src/README.md`.
- The jest suite must stay green; it gates every pipeline commit.
- Scripts anchor all paths at the repo root (`__dirname/..`) — keep it that
  way; there is no monorepo parent anymore.
- The CAP app itself is NOT built here — that is builder-cap2UI5's job. The
  app class `z2ui5_cl_app_read_odata` lives there (`src/srv/app/`), not in
  the core.
